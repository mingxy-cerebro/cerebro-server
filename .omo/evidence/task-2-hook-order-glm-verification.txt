## EVIDENCE 1: Hook Execution Order (from OpenCode binary decompilation)
================================================================================

Source: /home/dongx/.opencode/bin/opencode (ELF 64-bit, strings extraction)

The execution flow in SessionPrompt.createUserMessage (decompiled from binary):

```
STEP 1: Resolve user parts (files, agents, text)
        → vA = B.fn("SessionPrompt.resolveUserPart")
        → dA = yield*B.forEach($.parts, vA)  // resolve all parts

STEP 2: TRIGGER "chat.message" hook
        → yield*s.trigger("chat.message", {
             sessionID: $.sessionID,
             agent: $.agent,
             model: $.model,
             messageID: $.messageID,
             variant: $.variant
           }, {
             message: V,    // the UserMessage object
             parts: dA      // the resolved parts array
           })

STEP 3: Save message and parts to session store
        → yield*e.updateMessage(V)
        → for (T of Z) yield*e.updatePart(T)

STEP 4: Build system prompt
        → fj = [...bj, ...wj, ...IA?[IA]:[]]  // environment + system + skills

STEP 5: Call LLM provider process
        → The system.transform hook is triggered EARLIER, in Agent.generate():
           → yield*o.trigger("experimental.chat.system.transform", {model:A}, {system:b})
           → This happens when building the system prompt BEFORE the main loop

STEP 6: TRIGGER "experimental.chat.messages.transform" hook
        → yield*s.trigger("experimental.chat.messages.transform", {}, {messages: CA})

STEP 7: Convert to model messages via toModelMessagesEffect
        → VA.toModelMessagesEffect(CA, Z)
        → This is where ALL parts (including synthetic) become user messages
```

### ACTUAL EXECUTION ORDER:

In the main prompt loop (SessionPrompt):

```
1. chat.message           ← Plugin can add synthetic parts to output.parts
2. (save to store)
3. (build system prompt)
4. experimental.chat.messages.transform  ← Can transform full message history
5. toModelMessagesEffect  ← Converts ALL parts (incl. synthetic) to user messages
6. (LLM call with system + messages)
```

In Agent.generate (agent creation flow, SEPARATE from main loop):

```
1. experimental.chat.system.transform  ← Modifies system[] array only
```

### KEY INSIGHT: system.transform is NOT in the main prompt loop!

The `experimental.chat.system.transform` hook is called in TWO contexts:

CONTEXT A - Agent.generate (agent creation):
```javascript
let b = [he];  // initial system prompt
yield*o.trigger("experimental.chat.system.transform", {model:A}, {system:b});
// → b is now modified by plugins
// → Used as system messages for the agent creation LLM call
```

CONTEXT B - Main prompt loop (via Plugin hook registration):
```javascript
// In our plugin's index.ts:
"experimental.chat.system.transform": async (input, output) => {
    // This receives { system: string[] } and can append to output.system
    return wrappedRecallHook(input, output);
}
```

The main loop flow is:
1. **chat.message** fires → plugins can inject synthetic parts into the user message
2. **experimental.chat.messages.transform** fires → can transform full message history
3. **toModelMessagesEffect** → converts parts (including synthetic) to model-format messages
4. System prompt is built from: environment + skills + system (from system.transform)

## EVIDENCE 2: Synthetic Parts → User Message (from binary decompilation)
================================================================================

The `toModelMessagesEffect` function converts session messages to API-format messages.

From the binary, we can see how synthetic text parts are handled:

```javascript
// In SessionPrompt.resolveUserPart:
// All resolved parts get synthetic: true flag when they're auto-generated
{messageID: V.id, sessionID: $.sessionID, type: "text", synthetic: true, text: "..."}

// In the parts reduction:
hA = Z.reduce((T, tA) => {
    if (tA.type === "text") {
        if (tA.synthetic) T.synthetic.push(tA.text);  // synthetic tracked separately
        else T.text.push(tA.text);                      // real user text tracked separately
    }
    // ... files, agents, etc.
}, {text: [], files: [], agents: [], references: [], synthetic: []});
```

CRITICAL: The synthetic parts are STILL part of the same UserMessage (V).
They share the same messageID and sessionID.
When toModelMessagesEffect converts them, they become part of the USER role message.

This is confirmed by opencode-mem plugin behavior:
```javascript

    synthetic: true,
```

## EVIDENCE 3: GLM API Compatibility Analysis
================================================================================

### Why synthetic parts are safe for GLM models:

1. **Synthetic parts are in the USER message, NOT the system message**
   - The system prompt is built from a SEPARATE path (system[] array)
   - Synthetic parts go into output.parts of the chat.message hook
   - These parts belong to the UserMessage object
   - toModelMessagesEffect converts them to `{role: "user", content: [...]}`

2. **GLM API truncation behavior**
   - GLM models may truncate system[0] to a fixed length
   - But synthetic parts are NOT in system[0]
   - They're in the user message body, which is NOT truncated by GLM's system prompt handling

3. **Our Cerebro plugin's approach**
   - In `plugins/opencode/src/index.ts`:
     - `chat.message` hook → keywordDetectionHook (ingest user messages)
     - `experimental.chat.system.transform` hook → autoRecallHook (inject memory context to output.system)
   - The autoRecallHook writes to `output.system` (the system prompt), NOT to output.parts
   - This is CORRECT because output.system goes through system.transform
   - Memory context injected via system.transform IS subject to GLM truncation risk
   - But this is by design — it's the system prompt, not user message content

4. **opencode-mem/supermemory approach (DIFFERENT)**
   - They inject synthetic parts via chat.message → output.parts
   - These become user message content → NOT affected by system truncation
   - This is a VALID approach for GLM compatibility

### RISK ASSESSMENT:

| Injection Point | Goes To | GLM Truncation Risk |
|----------------|---------|---------------------|
| chat.message → output.parts | User message | ❌ NO RISK |
| system.transform → output.system | System prompt | ⚠️ POSSIBLE (if system[0] is too long) |
| messages.transform → messages | Full history | ❌ NO RISK (user messages preserved) |

## EVIDENCE 4: Cerebro Plugin Current Hook Registration
================================================================================

From plugins/opencode/src/index.ts (lines 148-176):

```typescript
return {
    "experimental.chat.system.transform": async (input, output) => {
        // 1. Locks mainSessionId
        // 2. Calls wrappedRecallHook → autoRecallHook
        //    → Injects memory context to output.system (SYSTEM PROMPT)
        // 3. Soul whisper: appends tool usage hints to output.system
    },
    "chat.message": keywordDetectionHook(...),
        // → Detects keywords for auto-capture
        // → Does NOT inject synthetic parts
    "experimental.session.compacting": compactingHook(...),
    "experimental.compaction.autocontinue": autocontinueHook(...),
    "tool.execute.before": soulWhisperToolTracker(...),
    "shell.env": ...,
};
```

### Flow for our plugin:
1. User sends message
2. chat.message fires → keywordDetectionHook runs (ingestion only, no injection)
3. system.transform fires → autoRecallHook injects memories to output.system
4. Soul whisper appends tool hints to output.system
5. System prompt (with memories) is sent to LLM as system message

### GLM COMPATIBILITY CONCERN:
Our memory context is injected to output.system (system prompt).
If GLM truncates the system prompt, memories could be lost.

SOLUTION (if needed): Switch to chat.message + output.parts injection
like opencode-mem/supermemory do, to put memories in user message instead.

## CONCLUSION
================================================================================

1. ✅ chat.message executes BEFORE system.transform in the main prompt flow
   - chat.message fires first (allows modifying user message parts)
   - Then messages.transform fires
   - Then toModelMessagesEffect converts everything to API format
   - system.transform is called separately for system prompt construction

2. ✅ Synthetic parts (from chat.message hook) are merged into the USER message
   - They share the UserMessage's messageID
   - They are NOT part of the system message
   - They are NOT affected by system[0] truncation

3. ⚠️ Our Cerebro plugin injects to output.system (via system.transform)
   - This IS subject to potential GLM system prompt truncation
   - Alternative: inject via chat.message → output.parts (like opencode-mem)
   - Current approach is architecturally valid but has GLM truncation risk

4. ℹ️ The hook execution order is:
   chat.message → messages.transform → toModelMessagesEffect → LLM call
   (system.transform is separate, called during system prompt construction)
