### Task 1: C2 — 正则编译缓存（LazyLock）

**Files:**
- Modify: `omem-server/src/ingest/extractor.rs:151-180` (calculate_quality_score内7个Regex)
- Modify: `omem-server/src/api/handlers/session_recalls.rs:514-520` (3个Regex)

**问题**: `calculate_quality_score()`每次调用编译7个正则，`session_recalls.rs`每次请求编译3个正则。高频路径，纯CPU浪费。

**方案**: 用 `LazyLock<Regex>` 提升到模块级/函数级静态变量。

- [ ] **Step 1: extractor.rs 正则缓存**

在 `calculate_quality_score()` 函数之前添加静态正则定义。将函数内部的7个 `Regex::new().unwrap()` 替换为引用静态变量：

```rust
use std::sync::LazyLock;
use regex::Regex;

static QUALITY_PATTERNS: LazyLock<Vec<Regex>> = LazyLock::new(|| {
    vec![
        Regex::new(r"[\u4e00-\u9fff]").unwrap(),      // CJK字符
        Regex::new(r"[.!?。！？]").unwrap(),             // 句末标点
        Regex::new(r"\b\d+\.?\d*\b").unwrap(),          // 数字
        Regex::new(r"(?:because|since|therefore|所以|因为|因此)").unwrap(), // 因果词
        Regex::new(r"(?:however|but|although|但是|不过|虽然)").unwrap(),    // 转折词
        Regex::new(r"(?:important|key|crucial|重要|关键|核心)").unwrap(),   // 重要标记
        Regex::new(r"(?:prefer|like|want|喜欢|想要|偏好)").unwrap(),        // 偏好词
    ]
});
```

然后在 `calculate_quality_score()` 内部替换 `Regex::new(X).unwrap()` 为 `&QUALITY_PATTERNS[i]`。

注意：先读取extractor.rs:151-180确认实际正则内容，上面的模式可能不完全准确，需要匹配实际代码。

- [ ] **Step 2: session_recalls.rs 正则缓存**

在 `session_recalls.rs` 中找到3个每次调用重编译的Regex（约514-520行），提取为模块级LazyLock：

```rust
static RECALL_FILTER_PATTERNS: LazyLock<[Regex; 3]> = LazyLock::new(|| {
    [
        Regex::new(r"...").unwrap(),  // 读取实际代码确认pattern
        Regex::new(r"...").unwrap(),
        Regex::new(r"...").unwrap(),
    ]
});
```

### Task 3: A4 — session_locks 清理

**Files:**
- Modify: `omem-server/src/api/server.rs:29` (DashMap声明)
- Modify: `omem-server/src/api/handlers/memory.rs:1328-1330` (使用位置)

**问题**: `session_locks: Arc<DashMap<String, Arc<Mutex<()>>>>` 永不清理，长期运行内存泄漏。

**方案**: 添加定期清理过期锁条目的逻辑。使用 `DashMap::retain()` 只保留最近活跃的条目。

- [ ] **Step 1: 在 memory.rs 中使用锁后标记时间**

在使用 session_locks 的代码（约line 1328-1330）之后，不需要额外操作。改用TTL方式：在每次使用时记录最后活跃时间，然后用 scheduler 清理。

- [ ] **Step 2: 添加清理逻辑**

在 server.rs 的 AppState impl 或独立函数中：

```rust
impl AppState {
    /// 清理超过1小时未使用的session锁
    pub fn cleanup_session_locks(&self) {
        // DashMap没有内置TTL，改为简单策略：
        // 保留最近N个session，清理其余
        // 或：每次获取锁时记录时间，定期清理过期
    }
}
```

**更简单的方案**: 不改DashMap结构，在LifecycleScheduler中定期清理：

1. 给 session_locks 的 value 改为 `(Arc<Mutex<()>>, Instant)` 或用新的wrapper
2. 或者更简单：直接在 scheduler 的 tick 中 `session_locks.retain(|k, v| /* 最近活跃 */)` 

**最简方案（推荐）**: 改 value 为 `(Arc<Mutex<()>>, std::time::Instant)`，每次使用时更新时间，scheduler 中 retain 24小时以上的。

**Files:**

**问题**: vault.rs 使用 SHA256 做密码哈希，太快容易被暴力破解。

- [ ] **Step 1: 添加 argon2 依赖**

在 Cargo.toml 中添加：
```toml
argon2 = { version = "0.5", features = ["std"] }
```

- [ ] **Step 2: 读取vault.rs现有哈希逻辑**

确认SHA256的使用方式，然后替换为Argon2：
```rust
use argon2::{Argon2, PasswordHash, PasswordHasher, PasswordVerifier};
use argon2::password_hash::SaltString;

fn hash_password(password: &str) -> Result<String, OmemError> {
    let salt = SaltString::generate(&mut rand::thread_rng());
    let argon2 = Argon2::default();
    let hash = argon2.hash_password(password.as_bytes(), &salt)
        .map_err(|e| OmemError::Internal(format!("hash error: {e}")))?;
    Ok(hash.to_string())
}

fn verify_password(password: &str, hash: &str) -> Result<bool, OmemError> {
    let parsed = PasswordHash::new(hash)
        .map_err(|e| OmemError::Internal(format!("parse hash: {e}")))?;
    Ok(Argon2::default().verify_password(password.as_bytes(), &parsed).is_ok())
}
```

- [ ] **Step 3: 迁移策略**

需要向后兼容已有的SHA256哈希。在验证时：
1. 先尝试Argon2验证
2. 如果格式不匹配（Argon2哈希以 `$argon2` 开头），fallback到SHA256验证
3. 验证通过后自动re-hash为Argon2
