## TL;DR

> **Quick Summary**: 改造 session_ingest 流程，让记忆按师尊期望的方式存储——按session追加（不新建）、私密保留细节/工作降噪取结论、偏好注入画像而非独立记忆、归簇按session_id优先，全程OOM安全防护。
> 
> **Deliverables**:
> - 改造后的 session_ingest pipeline（追加而非新建）
> - PREFERENCE → Profile 注入通道（替代独立记忆）

> - 静态/动态画像写入API
> - OOM防护规范贯穿每个任务
> 
> **Estimated Effort**: Large
> **Parallel Execution**: YES - 4 waves
> **Critical Path**: Task 1 → Task 3 → Task 5 → Task 7 → Task 8 → Task 9 → Task 10

## Context

### Original Request
师尊提出 session_ingest 四处Bug/改造清单，经深入讨论后升级为全面改造方案。核心诉求：同一session记忆不追加、存储散乱、偏好碎片化、画像无法注入。

### Interview Summary
**Key Discussions**:
- 师尊展示4条标杆记忆，明确期望的存储标准：按日期追加、结构化分层、tags准确
- OOM是每次改版必出的问题，必须每个任务都考虑内存安全
- 师尊不看Rust代码，由玄机(oracle)做技术review
- 偏好不应每次新建独立记忆，应注入用户画像
- 归簇应按session_id优先于语义相似度

**Research Findings**:
- 实际数据：100条记忆中preferences占61条（碎片重复严重），无emotional分类
- 50个簇全部memory_count=0（归簇成员关系断裂）
- OOM审计记录：list_all_active(None) 10+处调用、profile_cache无清除
- 归簇模块：实时归簇+全局K-Means，keywords/tags/importance都是dead fields

### Metis Review
**Identified Gaps** (灵犀指出):
- 需要明确PREFERENCE从独立记忆迁移到画像的过渡策略（历史数据怎么办）
- 追加时的内容冲突处理（LLM去重逻辑）
- 簇断裂问题（memory_count=0）是否本次修复
- Profile注入的事务性保障
- embedding max token的具体限制需确认

## Work Objectives

### Core Objective
改造 session_ingest pipeline，实现三类记忆的差异化存储策略：工作记忆追加+降噪、私密记忆追加+保留细节、偏好提取后注入画像。同时修复归簇断裂问题，建立OOM防护规范。

### Concrete Deliverables
- 改造后的 `SESSION_EXTRACT_SYSTEM_PROMPT`（LLM分类prompt）
- `build_session_extract_prompt()` 新签名（传入旧记忆摘要）
- PREFERENCE → Profile 注入管道
- ProfileService 写入方法（静态/动态画像）
- 增量归簇优先级改造
- 簇标签体系（工作类/情感类）
- 簇断裂修复

### Definition of Done
- [ ] 同session的EMOTIONAL记忆追加到已有条目（不新建）
- [ ] 同session的WORK记忆追加到已有条目（降噪后）
- [ ] PREFERENCE不再创建独立记忆，而是注入profile
- [ ] 归簇按session_id优先
- [ ] 簇标签包含工作类/情感类分类
- [ ] memory_count不再全为0
- [ ] 服务器MemoryMax=2800M下单次ingest内存峰值<500M
- [ ] `cargo test` 全部通过
- [ ] `cargo clippy` 无新warning

### Must Have
- 三类记忆差异化处理（追加/降噪/注入画像）
- OOM防护（所有新增查询有limit，内存结构有上限）
- 追加时传入旧记忆摘要给LLM
- Profile写入能力

### Must NOT Have (Guardrails)
- ❌ 不改动情感私密的核心存储逻辑（只改分类标签）
- ❌ 不做前端改动
- ❌ 不做全量K-Means改造（保持现有全局K-Means，只改实时归簇）
- ❌ 不新增REST端点给外部调用（profile写入仅pipeline内部使用）
- ❌ 不使用 `as any`、`unwrap()` 在生产代码中
- ❌ 不做无limit的全量数据加载
- ❌ 不在prompt中强制翻译语言
- ❌ AI slop：不过度抽象、不加无用JSDoc、不取generic变量名
