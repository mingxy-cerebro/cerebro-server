# OMEM v1.8.0 Ingest Pipeline 修复计划

**日期**: 2026-05-03
**优先级**: P1（数据质量问题）
**状态**: 待执行

---

## 问题诊断

### P2: 压缩垃圾数据被存入记忆

**现象**: 记忆ID `1632a532-b877-4de6-9f09-b9df29d9ca85`，内容为"完成DCP压缩#10，移除-161.6K数据，新增+12.8K摘要。"，这种compress工具的元数据被当成有价值的记忆存入。

**根因链**:
1. `extractor.rs::strip_envelope_metadata()` (lines 198-211) 只剥离 System Channel/Conversation info/Sender info 三种元数据，**不过滤** `[Compressed conversation section]`、`<dcp-message-id>`、`[system-reminder]` 等压缩伪影
2. `pipeline.rs::should_skip_content()` (lines 367-393) 需要 **3+个** 子串匹配才跳过消息，太宽松。一条只有1-2个压缩标记的消息会通过
3. `pipeline.rs::is_meta_operation()` (lines 363-365) 用 RegexSet，匹配1个就跳过，但它主要匹配工具命令(cargo build等)，不匹配压缩相关模式
4. noise filter 在 LLM 提取**之后**运行（Stage 6），如果 LLM 把压缩内容改写成看似正常的"事实"，regex 匹配不到

**影响**: 无价值记忆占用存储、污染搜索结果、浪费LLM token

### P3: 同session跨时段对话不合并记忆

**现象**: 同一个session（如 ses_226860b0），早上聊天和晚上聊天各自产生独立的记忆，reconciler不会将后者合并到前者。

**根因链**:
1. `reconciler.rs::reconcile()` 接收 `session_id` 参数 (line 53) 但**完全不用于搜索**
2. `gather_existing()` (lines 459-542) 通过全局向量搜索和FTS搜索查找相似记忆，**没有 session_id 过滤**
3. `build_reconcile_prompt()` (`prompts.rs` lines 27-89) 展示现有记忆格式为 `[int_id] (category: {}, age: {}) l0_abstract`，**不包含 session_id**
4. LLM 做 merge 决策时不知道哪些记忆来自同一 session，无法优先合并
5. `session_id` 在整个 reconciler 中是 **write-only**（只写入新 Memory，不用于查询）

**影响**: 同一用户的同一session积累大量碎片化记忆，应该合并的信息散落在多条记忆中

## 修复方案

### Fix A: 加强压缩/工具输出过滤（修 P2）

#### A1: extractor.rs 增加压缩伪影剥离

**文件**: `omem-server/src/ingest/extractor.rs`

**修改**: 在 `strip_envelope_metadata()` 函数 (lines 198-211) 中增加以下模式的剥离：

```rust
// 在现有三个 strip 规则之后添加：
// 4. [Compressed conversation section] 块
// 5. <dcp-message-id>...</dcp-message-id> 标签
// 6. [system-reminder] 块（含多行内容直到下一个空行或标签）
// 7. [Category+Skill Reminder] 块
// 8. <!-- OMO_INTERNAL_INITIATOR --> 注释
```

**实现要求**:
- 用 Regex 或行级扫描，删除匹配行
- 处理多行块（[system-reminder] 通常跨多行直到下一个分隔符）
- 不要删除消息正文，只删除系统伪影标签

#### A2: pipeline.rs 降低 should_skip_content 阈值

**文件**: `omem-server/src/ingest/pipeline.rs`

**修改**: `should_skip_content()` (lines 367-393)，将阈值从 **3** 降低到 **1**：

```rust
// 当前: if count >= 3 { return true; }
// 改为: if count >= 1 { return true; }
```

**理由**: 消息中只要出现1个系统模式（如 `session_compress`、`compression #`、`dcp-message-id`），就说明这是系统生成内容，不应该被提取。

#### A3: noise.rs 增加压缩相关 noise 模式

**文件**: `omem-server/src/ingest/noise.rs`

**修改**: 在 `build_regex_patterns()` (lines 82-133) 的 patterns 列表中添加：

```rust
// 压缩元数据
r"(?i)完成DCP压缩#\d+",
r"(?i)移除-?[\d.]+K?数据",
r"(?i)新增\+?[\d.]+K?摘要",
r"(?i)compressed \d+ messages?",
r"(?i)compression #\d+",

// 工具调用残留
r"(?i)background (?:task|output)",
r"(?i)task_id.*bg_",
r"(?i)playwright_browser_\w+",
r"(?i)lsp_diagnostics",
r"(?i)ast_grep_(?:search|replace)",
```

### Fix B: 同session记忆合并（修 P3）

#### B1: reconciler.rs 增加 session_id 感知的合并优先级

**文件**: `omem-server/src/ingest/reconciler.rs`

**修改1**: 在 `gather_existing()` (lines 459-542) 中，除了现有的全局向量/FTS搜索，**额外**查询同 session_id 的记忆：

```rust
// 在 gather_existing() 的搜索逻辑中添加：
// 1. 先按 session_id 查询该 session 已有的所有 memory（用 LanceDB 的 filter）
// 2. 将这些 memory 加入 existing_candidates，标记 origin = "same_session"
// 3. 同 session 的记忆即使向量相似度较低也要包含（设置较低阈值如 0.3）
```

**修改2**: 在 `build_reconcile_prompt()` 中，给同 session 的记忆加标记：

```
// 现有格式: [1] (category: preferences, age: 2h) 用户喜欢...
// 新格式:   [1] (category: preferences, age: 2h, same_session) 用户喜欢...
```

这样 LLM 在做 merge 决策时能看到同 session 的记忆，优先合并。

#### B2: reconciler.rs 的 reconcile() 中同session合并逻辑

**文件**: `omem-server/src/ingest/reconciler.rs`

**在 reconcile() 的 LLM 决策之前（Stage D 之前），插入快速路径**：

```rust
// 快速路径：同 session 的记忆如果 l0_abstract 语义重叠度 > 0.5，直接 MERGE
// 避免每次都走 LLM，节省 token
for fact in &facts {
    for existing in &same_session_memories {
        if semantic_overlap(fact.l0_abstract, existing.l0_abstract) > 0.5 {
            // 直接标记为 MERGE，跳过 LLM reconciliation
        }
    }
}
```

### Fix C: source=session_compress 的记忆默认降级（防御性措施）

**文件**: `omem-server/src/ingest/pipeline.rs`

**修改**: 在 admission control 阶段 (Stage 7, lines 208-245)，对 `source == "session_compress"` 的记忆施加额外过滤：

```rust
// 如果记忆的来源是 session_compress：
// - 要求 importance > 0.7 才准入（默认0.5的直接拒绝）
// - 或者要求包含实质性的用户偏好/决策/事实
// - 纯元数据（"压缩#N"、"移除XKB"等）直接拒绝
```
